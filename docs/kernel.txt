Spike
 2026

 Run a 200-billion parameter model on a machine that can't hold it in RAM. That's not a demo. That's what
 this does.

 The OS treats virtual memory like it was designed for code, not for weights. It pages things in on
 demand — after the GPU asks, not before. That means the GPU stalls while it waits for disk. Unacceptable
 when loading a 512MB block from NVMe takes ~100ms.

 Spike sits below the inference engine. It's a single function — pager_get(block_id) — that replaces
 direct memory reads. When the GPU asks for a weight block, Spike already knows what's next. It
 loaded it in the background while the GPU was still computing, so the block is already in RAM. No stall.

 The secret is a Markov predictor trained from access traces. It learns the model's access fingerprint:
 "if the GPU just touched block 10, there's a 91% chance it's about to ask for block 11". When that
 prediction fires, a background thread loads the block before it's needed.

 Benchmark on simulated Qwen3 14B-like access patterns (28 layers, 64 experts):

 ```
   baseline (LRU):    83.4% hot rate,   16.9 tokens/s,  51,226μs stall/token
   markov predictor:  86.0% hot rate,   19.6 tokens/s,  43,109μs stall/token

   speedup: 1.16x
   top-2 prediction accuracy: 91.9%
 ```

 On real Apple Silicon M1 with 16GB unified memory running Qwen3-32B-MLX-4bit (17.5 GB weights):

 ```
   tokens/s:  0.133
   wall time: 401s for 50 tokens
   peak RSS:  2.64 GB
   peak Metal: 1.13 GB   (vs 18.43 GB without paging — OOM)
 ```

 The predictor's background prefetch thread hides NVMe latency while the GPU computes. Fewer disk reads
 also means less energy: -15.8% I/O (557 GB saved) on the benchmark = -111 mWh.

 Built in C11. Zero external dependencies. pthreads, POSIX mmap. Works on Linux and macOS.

 ```
   pager_get(&p, block_id)   ← drop-in replacement
   pager_release(&p, block_id)
 ```

 The file format, the predictor, the pager — everything is under 3,000 lines total. A new contributor can
 read it in an afternoon.

 Build and run:

   make
   ./benchmark 512 256      # key result: 1.16x speedup
   ./synthetic 128          # 128-token trace generation
   ./analyze synthetic.wptr 18   # transition matrix analysis
